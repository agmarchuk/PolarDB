Введение
Концепция подхода
Задача проекта заключается в том, чтобы создать эффективные средства представления данных в виде классов, файловых и других построений. Причем данные могут быть распределены по файловой системе или по сети компьютеров. Способ достижения эффективности - опора на строгую типизацию хранимых и обрабатываемых данных, минимизация чтений из файлов и записей в файлы, минимизация объемов активных данных.

Общее представление о структуризации
Структурное значение (structured value, поляровское значение, поляровская структура) - древовидное построение (значение), интерпретируемое в соответствии с заданным типом.

Тип (Type, поляровский тип) - древовидное построение, задающее интерпретацию для структурных значений. Может быть выражено как структурное значение.

Типы могут быть примитивными или атомарными (primitive types) или конструируемыми (constructed types), есть также строки (strings), которые условно можно включить в набор примитивных.

Примитивные типы: boolean, character, integer, longinteger, real и, кроме того, тип none - множество значений, не содержащее ни одного элемента.

Конструируемые типы:

запись (record) - фиксированный набор типизированных полей. Есть поля 0, 1, ... n-1, каждое из которых представляет значение заданного в определении записи типа. Количество полей - фиксировано, типы полей - фиксированы.
последовательность (sequence) - упорядоченный набор, состоящий из неопределенного (ноль или более) числа однотипных элементов.
объединение (union) - значение, состоящее из тега и подзначения. Тег (динамически) определяет вариант типа для подзначения. Теги нумеруются, начиная с 0.
Строки: sstring - соответствует понятию (системной) строки в объектно-ориентированном программировании. Отличается от последовательности символов тем, что длина объекта sstring меньше соответствует нормам используемой системы программирования. В .NET это больше 1 миллиарда символов.

Пакет определений PType - дает определения общих представлений типов и определения объектов PType. Объекты класса PType - объекты C#, задающие тип (поляровский тип).

Структурные значения корректно интерпретируются только в контексте (поляровского) типа. Само значение типа не содержит, тип надо формировать отдельно или где-то его прочитать. Объектом обработки является формально или неформально составленная пара {v, t}, где v - само значение, t - значение его типа.

Структурные значения имеют несколько вариантов представления: объектное, байтовая сериализация, текстовая сериализация. Кроме того, возможно использовавние JSON и встроенное представление (сейчас не реализовано).

Основные разделы библиотеки
Общий раздел: типы, базовые структуры, сериализации
Раздел размещен в пространстве имен Polar.DB. Класс, определяющий типовый объект - PType. Его конструктор позволяет рекуррентно задавать типы для иерархических объектных значений. Например типовое определение,

    PType tp_rec = new PTypeRecord(
        new NamedType("имя", new PType(PTypeEnumeration.sstring)),
        new NamedType("возраст", new PType(PTypeEnumeration.integer)),
        new NamedType("мужчина", new PType(PTypeEnumeration.boolean)));
задает запись с тремя полями, строкового, целого и булевского типов.

Соответственно,

    PType tp_seq = new PTypeSequence(tp_rec);
задает поляровский тип последовательности значений указанного типа.

Поляровский тип необходим для корректной интерпретации структурных значений. Структурные значения могут иметь разные представления: объектное, текстовое, бинарное.

Рассмотрим объектное представление значений. Значения типов логическое, байт, символ, целое, длинное целое, вещественное, строковое представлются объектами (object) соответствующего типа byte, char, int, long, double, string. Значение записи представляется массивом объектов, в котором элементы массива соответствуют элементам записи. Значение последовательности представляется массивом объектов, в котором элементы массива соответствуют элементам последовательности. Значение объединения состоит из пары: целочисленное значение тега и значение подобъекта. Пара выполняется как массив из двух элементов (объектов). Приведенный пример иллюстрирует задание объекта типа последовательности записей:

    // фрагмент теста программы на C#
    object seq_value = new object[]
    {
        new object[] { "Иванов", 24, true },
        new object[] { "Петрова", 18, false },
        new object[] { "Пупкин", 22, true }
    };
Другие представления поляровских объектов задают сериализации - текстовая и бинарная. Тестовая сериализация, это представление объекта в виде текста. Синтаксис представления следующий: атомарные объекты изображаются традиционно, как в C#, записи перечисляют элементы, использую фигурные скобки, последовательности группируют элементы используя прямоугольные скобки. Например, данный текст может использоваться как данное или выводиться как результат:

    [
        { "Иванов", 24, true },
        { "Петрова", 18, false },
        { "Пупкин", 22, true }
    ]
пробелы и переносы не существенны.

Превращение объекта в сериализованный текст выполняется методом TextFlow.Serialize(), а обратное действие методом TextFlow.Deserialize(), например:

    TextWriter tw = new StreamWriter(stream1);
    TextFlow.Serialize(tw, seq_value, tp_seq);

    ...

    TextReader tr = new StreamReader(mstream2);
    object db = TextFlow.Deserialize(tr, tp_seq);    
То же самое можно делать, используя байтовую сериализацию. Представление структуры данных в виде последовательности байтов выглядит следующим образом:

примитивные объекты и строки укладываются в в поток байтов средствами BinaryWriter и читаются оттуда BinaryReader
записи укладываются подряд идущими элементами
последовательности первые 8 байтов используют для указания long количества элементов последовательности, а потом подряд укладываются сами элементы
объединения записывают один байт - номер варианта, а потом укладывается вариант подзначения, для этого требуется ноль или более байтов.
Байтовые сериализация и десериализация выполняются аналогичным образом, только вместо текстового ридера/райтера используются бинарные:

    BinaryWriter bw = new BinaryWriter(stream3);
    ByteFlow.Serialize(bw, seq_value, tp_seq);

    ...

    BinaryReader tr = new BinaryReader(mstream4);
    object db = BinaryFlow.Deserialize(tr, tp_seq);    
Таким образом, сформирован базис представлений и действий, позволяющий порождать структурированные объекты, конвертировать их в тексты и потоки байтов, напр. в файлы, читать из текстов или файлов. Базис позволяет использовать и преобразовывать их в объектном представлении. Важно отметить, что в силу рекуррентности текстового и байтового представлений, начало подобъекта сериализованного объекта можно фиксировать через позицию в файле и считывать этот подобъект как независимый. Для байтовых разверток можно и перезаписывать подобъекты на новые значения, но это только если они относятся к примитивным или это записи примитивных объектов. Это потому, что размер подобъекта не должен меняться при перезаписывании.

Этот базис является лишь первичным звеном платформы структуризации. Чем он плох? Плох тем, что база данных должна вся загружаться в оперативную память, что существенно ограничивает возможности как в части общего объема используемых данных, так и в оперативности "первого" реагирования за какого-нибудь запрос.

Общий раздел: последовательности
Последовательность являются той стуруктурой, которая позволяет для базы данных говорить "много". Много чего? Много элементов, это может быть список, множество, мультимножество, массив. Последовательность элементов в парадигме Поляра, это последовательность, т.е. нуль или более элементов какого-то определенного типа. Когда типом элемента является запись, то можно говорить о таблице, в которой поля записи формируют столбцы, а значения - рядки данных таблицы.

В большинстве СУБД, таблицы имеют свойства "на все случаи жизни". В поляре, это не так. Дело в том, что слишком универсальная таблица будет иметь большие накладные расходы, а все свойства могут не понадобиться. В качестве примера, рассмотрим заполнение данными таблицы. Если рассматривать таблицу просто как накопитель информации, т.е. множества записей, то поляровское решение будет в 10-15 раз более быстрым, чем решение, построенное на базе универсальной СУБД, напр. SQLite. На таблицах со сложной функциональностью разница будет менее значительная или ееможет не быть совсем. Этот диапазон возможностей заполняют разные по логике и характеру работы решения библиотеки Polar.DB.

Для характеризации решений, подумаем о способах использования последовательностей в приложениях. Особенно - больших последовательностей. Последовательность может быть сначала сформирована (построена, заполнена данными), а потом использована в качестве источника данных или последовательность может быть заполняема и использована одновременно. Что это означает? Пусть seq - последовательность, seq.Add(v) - добавление элемента к последовательности, seq.Get(get params) - выборка из последовательности. Причем Add меняет состояние последовательности, а Get - не меняет. Первый режим использования:

инициирование seq
seq.Add(v1), seq.Add(v2), ...
доделываение seq
а = seq.Get(..), b = seq.Get(..), ...
Второй режим:

инициирование seq
seq.Add(v1), а = seq.Get(..), seq.Add(v2), ...
b = seq.Get(..), ...
Во втором режиме нет доделывания. Может и есть, но оно встроено либо в Add, либо в Get. Если доделывание не нужно, то разнины между режимами нет. Если нужно, то она есть и возможно - серьезная. Заметим, что в первом режиме последовательность, находясь в состоянии приема данных, не обязана поддерживать возможность их использования, использование будет готовиться доделыванием. А в состоянии использования, состояние последовательности не меняется.

Откуда берется "доделывание"? В основном, это построение или коррекция индексов. Операция Get, при всей своей первичной простоте, часто требует дополнительной относительно собственно последовательности информации,эта информация называется индексом. И вот индексы могут довольно сильно изменяться при каждом (!) выполнении Add. Поэтому разовое построение индексов в "доделывании", часто предпочтительнее многократному, выполняемому после Add или перед Get. Эта проблема может быть несколько нивелирована алгоритмическими или системными средствами, но полностью устранена быть не может. Вот почему имеет смысл различать разные режимы использования.

Последовательности поляра формируются на основе класса UniversalSequenceBase. Этот класс генерирует значение при указании поляровского типа элемента и стрима как среды жизни последовательности.

    public UniversalSequenceBase(PType tp_el, Stream media)
У этого класса две задачи: быть базовым классом для более сложных построений и, вторая, иметь нужные методы для этих самых более сложных построений. В любом случае - это последовательность. Последоватльность, которую можно создать через конструктор, можно очистить, можно добавлять элементы, можно прочитать (просканировать) поэлементно. Последовательность представляет из себя бинарную сериализацию на какой-то стрим, обычно на файл. Соответственно, у элементов последовательности есть свои координаты: offset - смещение в числе байтов относительно начала стрима. Зная офсет можно прочитать (десериализовать) любой элемент последовательности и это - быстрая операция. Откуда можно узнать офсет? Проще всего он вычисляется для последовательностей, длина элементов которой фиксирована, тогда это будет 8 + i * elen. Где i - номер элемента в последовательности. Если длина элементов не фиксирована, офсет элемента возвращается при добавлении в последовательность long off = seq.AppendElement(e). Таким образом, можно где-то сохранить офсет и тогда им можно будет воспользоваться позже. Офсеты часто сохраняют также в последовательности, которая и становится индексной.

В универсальной последовательности есть методы, которые работают только для последовательностей с фиксированным размером элементов. В основном именно они являются "строительным материалом" индексных построений. Это упоминавшиеся чтение и запись по номеру элемента и сортировка.

Кроме универсальной последовательности, есть еще другие варианты, их обзор будет несколько позже. Главное, что эти варианты добавляют какие-то функциональности к последовательности. Эти новые функциональности даются "не бесплатно" и те функциональности, которые уже были, выполняются хуже (медленнее). Вот почему в библиотеке есть и более простые последовательности и более сложные.

Рассмотрим функциональности, связанные с последовательностями, разобъем их по группам.

Базовые средства:

new - Создание последовательности или подключение к последовательности;
AppendElement, Count - Добавление элементов, выдача размера;
Clear, Flush, Refresh, Close - Очищение, сброс буферов, разогрев, закрытие;
ElementValues, Scan - Выдача потока значений элементов, сканирование элементов;
Построения (доделывания)

Build построение
==============

Простая структура универсальной последовательности хорошо подходит к следущему варианту использования: последовательность инициируется (заводится или очищается), заполняется потоком значений через метод

    seq.Load(IEnumerable<object>);
а потом используется в вариантах работы с потоком значений или сканирования. К этому добавляются одиночные добавления элементов. Кроме этого, простая структура позволяет фиксировать позиции элементов в виде длинных целых, а по позциции элемента, можно элемент читать и можно изменять если его размер фиксирован.

В позиции можно углубляться до поля произвольного уровня. Как это делать. Сейчас этот момент не поддержан библиотекой. А как это должно быть? Пусть loc есть локация элемента типа tp. У элемента могут быть подэлементы, у последовательности - элементы последовательности, у записи - поле записи, у объединения - значение данного варианта. Где-то были соответствующие формулы. Сейчас посмотрю.

Насчет углубления. Здесь нужно вводить понятие позиции, локации и т.д. Тогда универсальная последовательность может буть носителем этих построений. Что-нибудь вроде:

    Location loc = new Entry(seq, off, tp);
А потом делать уточнения:

    var loc1 = loc.Field(1);
    object ob = loc1.Get();
    ...
    loc1.Set(ob1);
Что-то неубедительно...

Тогда поледовательность будет наследоваться от поляровского объекта, можно привязать сериализацию и десериализацию. Можно и не делать своего объекта, он получается, как

    Location myloc = new Entry(this, 0L, tp_obj);
Да, именно так!

Может надо по-новой рассмотреть вопрос с ячейками и входами (entry).

============ Следующий момент - некоторая иерархия решений по последовательностям. Итак, получится: общий класс PolarObject. С ним можно делать что хочешь. Потом - базовый класс универсальной последовательности. Потом - последовательность уничтожаемых элементов. Здесь - не факт. Можно спокойно выделить признак уничтоженности и уничтожать элементы. Но что это семантически? Набор элементов, которые можно сканировать, подвергать вычислению предикатов и функций. Как-то так...

Следующий уровень уже связан то ли с моделью работы с данными, либо с индексами. Почему с индексами? Непонятно... Допустим, мы делаем key-value хранилище. Ключи - уникальны, значение может быть null. Модель может быть инкрементная. Или с уничтожением. Инкрементная схема - только добавление элементов. Попадание в уже существующий ключ заменяет значение, связанном. Инкрементная схема действует так: вместе с key-value парой записывается отметка времени, записи с одинаковыми ключами факторизуются и оригиналом считается запись, с более поздней отметкой времени. Для того, чтобы иметь возможность удаления значения по заданному ключу, на значениях должен быть определен предикат IsDeleted, истинность которого обозначает, что значение "уничтожено" и должна быть схема, напр. в виде константы для порождения IsDeleted-значения.

Чем интересна схема? Она потенциально дает возможность поддерживать распределенные key-value хранилища. Идея здесь следующая. Каждый банк распределенного хранилища "работает" самостоятельно. Это ознаает, что можно записывать новые пары в любой, напр. ближайший, банк и читать (делать запросы на выборку) можно также из ближайшего банка. Но среди банков должна существовать система постепенной синхронизации, при которой банки обмениваются информацией о том где находится оригинал той или иной записи. Это может быть полузапись вида {key, timestamp, bank}. Пользуясь этой информацией, по запросу на выбоку по ключу, сервер определяет банк, к которому надо сделать запрос и получить оттуда значение. Понятно, что эта схема поддерживает eventual consistency (согласованность в конечном итоге) и результат иногда может быть "устаревшим". Но через фиксированный промежуток времени, при отсутствии новых изменений, по запросу всегда будет выдаваться нужный результат.

Возникает вопрос - надо ли вводить опорную последовательность? Опорная последовательность, это последовательность айтемов (записей), снабженная индексами и некоторой логикой работы с элементами последовательности. Логика 1 (BearingPure): просто элементы без искажения. Логика 2 (BearingDeletable): элементы, не помеченные как уничтоженные. Для этой логики действительны ВСЕ элементы кроме помещенных как deleted. Это касается и сканирований и поиска и выборок. Дополнительными операциями являются DeleteItem. Логика 3 (BearingTimestamped): это элементы, у которых отметки времени самые поздние среди элементов с одинаковым ключом.

Опорные (bearing) последовательности (таблицы) представляют собой последовательности элементов заданного типа, снабженные индексами. Тогда после наполнения (Load) последовательности, запускается построение (Build) индекса. И после построения, такой индекс уже не меняется также как и опорная таблица. Этот вид индексов называется неизменяемыми (immutable). Более сложные построения - изменяемые индексы, в них индексы изменяется в соответствии с изменениями опорной (и только опорной) последовательности. Происходит это по событиям "добавить элемент" и "уничтожить элемент".

Массив фиксирует значения элементов в объектном представлении для равномерной сетки точек, начиная с первого и кончая последним элементов. Естественно, построенный массив должен быть разреженный и помещаться в ОЗУ (выделенный ресурс ОЗУ). Эта библиотека предназначена для того, чтобы создавать пулы байтовых потоков (streams), управлять кешированием, в дальнейшем управлять избыточностью в данных, контролем и восстановлением данных.

В частности, дескриптор состоит из 13 элементов. Поток (PagedStream) начинается на другм потоке, который называется опорным. Вначале потока идет т.н. голова потока, поляроский тип головы приведен в статическом объекте TpHead. После головы идет начальная часть потока байтов, т.н. beginblock. Таким образом, каждый поток начинается на другом потоке и свободно развивается за счет ресурсов страничной памяти. Этот первый поток можно использовать для создания нескольких следующих, но небольшого их количества поскольку он имеет право развиваться только за счет голов других потоков и не переходить на другую страницу. Можно точно подсчитать сколько можно создать потоков на основе первичного. Но сейчас это не важно, один-два-несколько можно...

Реальная предлагаемая конструкция задания системы сравнения многократно проговаривалась: задаются два метода - ключевая функция и компаратор. Каждый из методов может быть null, но не оба. Логика совместного использования ключевой функции и компаратора такова:

int compare(e1, e2) => key(e1) < key(e2) ? -1 : (key(e1) > key(e2) ? 1 : equ(e1, e2) ); 
Возможно, надо equ свести к значениям -1, 0, 1.

В общем случае, надо говорить о согласованности или несогласованности ключевой функци и компаратора. Согласованность будет если для всех e1, e2 таких, что key(e1) < key(e2), компаратор дает такое же соответствие. В любом случае, синтетический компаратор, приведенный выше, согласован с ключевой функцией по определению.
