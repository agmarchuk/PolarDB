Описание библиотек
PolarDB состоит из ряда библиотек:

Polar.DB - базовая библиотека системы поляровской структуризации;
Polar.Cells - ячейки для хранения структурированных данных, средства формирования ячеек, средства доступа к этим данным;
Polar.CellIndexes - индексные построения для последовательностей и таблиц, вкладываемых в ячейки;
Polar.PagedStreams - классы, предназначеные для формирования неограниченного множества потоков (Streams);
Polar.Datanode - решения для распределенных хранения и обработки данных;
Polar.TripleStore - готовое решение для хранилища триплетов. Кроме того, есть набор тестовых задач и простых решений, построенных на библиотеке PolarDB, этот набор пока сведен к проекту GetStarted в котором каждая задача реализуется в виде своего Main() модуля, в некоторых случаях к модулю есть дополнительные определения классов.
Polar.DB Базовые средства
Поляровские типы присутствуют в программе в виде объектов класса PType и его производных PTypeRecord, PTypeSequence и PTypeUnion. Поляровские структурные объекты, которые являются древовидными построениями, интерпретируются только в контексте типа, соответственно при обработке, явно или неявно собирается пара {v, t}, гда v - структурное значение, t - тип этого значения. Сам тип представляет собой древовидное построение, к которому также применима поляровская структуризация. Соответственно, есть (константный) объект "тип типов" TType.

Типовая (ударение на первый слог) система Поляра содержит некоторое (отладочное) средство для визуализации структур. Текстовый вариант значения получается применением к значению класса PType метода string Interpret(object v), здесь v - объектное представление структурированного значения.

Следующая группа действий, определяемая в Polar.DB, это сериализации. Сериализация - это представление структурного значения в виде последовательности каки-то "букв". Десериализация - обратный процесс, т.е. получение объекта из цепочки букв. Есть две пары сериализация/десериализация: текстовая и бинарная. При текстовой сериализации, буквами являются символы юникода, при бинарной - байты. Модули TextSerialize и BinarySerialize дают процедуры прямого и обратного преобразований. Существенной чертой сериализаций/десериализаций является рекуррентность процессов преобразования. Это означает, что проведение сериализации структурного значения выполняется через сериализацию его элементов с добавлением связующих элементов. При этом, у каждой серии "букв" есть конкретный адрес, для байтов - это смещение (offset) в потоке. Таким образом, если находить начало серии и начинать чтение (десериализацию) с этой точки, мы будем получать копию записанного там структурного объекта.

Рассмотренные средства дают возможность провдить обработку такого вида:

PType tp = ...
object v1 = new object[] { ... };
Stream stream = File.Open( ... );
BinaryWriter bw = new BinaryWriter(stream);
...
// Где-то сохраняем позицию записи
long offset1 = stream.Position;
// Записываем значение в поток
Serialize(bw, v1, tp);
...
// Теперь когда значение понадобилось, установим позицию чтения и прочитаем
stream.Position = offset1;
BinaryReader br = new BinaryReader(stream);
object v2 = Deserialize(br, tp);
// значение v2 будет таким же, как v1.
Эта схема показывает как отдельный Stream (файл) или множество стримов можно использовать как некоторую память в которую можно положить структурное значение, а потом из этого места неограниченное число раз его брать. Схема выглядит громозкой, но она весьма эффективна и универсальна, а реально она используется через классы и методы более высокого уровня для конкретных частных случаев.

Как раз таким частным (и часто используемым) частным случаем является последовательность элементов определенного типа. Последовательность элементов возникает как задача накопления, хранения и обработки данных и при построении дополнительных структур (индексов), которые делают доступ к данным эффективным. Последовательность элементов одного типа можно задать классом UniversalSequenceBase параметрами конструктора которого являются тип элемента и Stream - среда бинарной реализации. Определив тип записи и создав объект класса UniversalSequenceBase, мы получим вполне работоспособную базу данных типа хранилища последовательности записей определенного вида:

    PType tp_el = new PTypeRecord(
        new NamedType("id", new PType(PTypeEnumeration.integer)),
        new NamedType("name", new PType(PTypeEnumeration.sstring)),
        new NamedType("age", new PType(PTypeEnumeration.real)));
     Stream media = File.Open(...);
     var ubs = UniversalSequenceBase(tp_el, media);        
Мы можем очищать последовательность, заполнять ее данными, можем добавлять отдельные элементы, читать элементы с установленной позиции, в определенных ситуациях, писать элементы или их части. Можно сканировать последовательность, т.е. последовательно читать элементы с начала и до конца. Существенным моментом является фиксированность или нефиксированость размера элемента. Структурнное значение обладает фиксированным размером если фиксированным размером обладают все элементы этого значения, простые типы почти все обладают фиксированным значением кроме строки. Разница в том, что если элемент фиксированного размера, то можно легко найти в считать i-ый элемент через простое вычисление офсета этого элемента:

offset = start + i * elementsize;
Для нефиксированного размера элементов, позицию элемента в потоке также можно определить, но только через сканирование, что неприемлемо долго для большинства случаев.

Еще одно действие задается только для последовательностей фиксированного размера. Это сортировка. Метод

public void Sort32(Func<object, int> keyFun);
выполнит сортировку последовательности относительно ключевой функции, заданной в аргументе. Функция - 32-разрядная, что ограничивает ее возможности для ряда случаев, есть аналогичный метод Sort64, в котором указывается 64-разрядная функция.

Наличие сортировки позволяет преодолеть трудности прямого доступа для последовательностей с нефиксированным размером элементов. Делается это по следующей схеме: кроме основной (опорной) последовательности, параллельно создается последовательность пар {key, offset} значений ключа и офсетов элементов, которые содержатся в опорной последовательности. Эта последовательность (индексная последовательность) имеет фиксированный размер элементов и может быть отсортирована по значению ключа. После этого, методом деления пополам (дихотомии), нужный для заданного ключа элемент получается за логарифимическое относительно размеров последовательности время, выбирая из этого элемента офсет, мы имеем начальную позицию элемента в основной (опорной) последовательности и, соответственно, можем сделать выборку найденного элемента.

Реализация такого подходя имеет ряд алгоритмических и технологических тонкостей, поэтому сведена к основным случаям в этой же библиотеке Polar.DB:

IndexKey32Comp
IndexView
IndexKey32CompVector
При определении универсального ключевого индекса IndexKey32Comp задаются:

генератор потоков, задающий носитель данных индекса
опорная последователности, которая индексируется
предикат применимости индекса к данному элементу
хеш-функция, заданная на элементах опорной последовательности
сравнитель (Comparer), сравнивающий элементы опорной последовательности.
Индекс работает следующим образом: элементы последовательности упорядочиваются сначала в порядке, задаваемым хеш-функцией, вторичный порядок задается компаратором. Хеш-функция или просто функция сортировки не всегда является "хеш", а компаратор может быть задан null, что означает его отсутствие. Определенный индекс может быть построен (Build), к нему можно обратиться с запросом обо всех элементах, соотвествующих некоторому образцу. На образце вычисляется значение хеш-функции и применим компаратор. Соответствующие образцу элементы, это элементы последовательности, значение хеш-функции на которых совпадает с заданным и сравнение компаратором дает 0.

Индекс IndexView создается также указанием среды хранения, опорной последовательности, применимости и компаратора. Соответственно, определяются элементы, которые при сравнении с образцом дают 0.
